# Optimization Framework

Небольшой фреймворк для итеративной градиентной оптимизации на Python. Идея в том, что сам оптимизатор (`MasterOptimizer`) ничего не знает о конкретных алгоритмах — он просто склеивает между собой набор взаимозаменяемых компонентов: чем обновлять параметры, как выбирать шаг, когда остановиться и нужна ли регуляризация. Поэтому добавление нового метода (скажем, ещё одного update rule) не требует трогать сам оптимизатор.

Помимо самого оптимизатора в репозитории лежат тестовые функции и лоссы для ML-моделей (на них всё это можно гонять и сравнивать), а также контейнеры для результатов бенчмарков.

## Как устроен `MasterOptimizer`

`optimize(x0)` — основной цикл. На каждой итерации:

1. через `gradient_estimator.estimate(objective, x)` получаем градиент в текущей точке, прогоняем его через `regularizer`;
2. собираем `OptimizationState` (текущие `x`, значение функции, градиент, его норма) и проверяем все `stopping_criteria` — если хоть один сработал, выходим с соответствующим `stop_reason`;
3. если не вышли — `step_strategy` отдаёт размер шага, `update_rule` превращает градиент в направление обновления, делаем шаг;
4. записываем итерацию в историю и обновляем лучшую найденную точку (`best_x`/`best_f`, не обязательно совпадает с финальной).

Траектория (`history.trajectory`) сохраняется только если размерность задачи не больше `store_trajectory_if_dim_leq` (по умолчанию 2) — иначе она бы просто раздувала память без особого смысла.

Счётчики обращений к функции (`function_calls`, `gradient_calls`, `samples_seen`) живут не в оптимизаторе, а в самом `Objective` — это сделано специально, чтобы можно было считать их честно независимо от того, кто и как дёргает функцию (сам оптимизатор, backtracking line search и т.п.).

## `Objective`

Базовый абстрактный класс для всего, что оптимизируется — что тестовая функция, что лосс модели. Наследник должен реализовать только `_compute_value` и `_compute_grad`; всё остальное (счётчики, `tags` с описанием свойств функции типа `"convex"` / `"multimodal"` / `"ill-conditioned"`) уже есть в базовом классе. `indices` во всех методах опциональны — если их передать, считаем не по всей выборке, а по батчу.

## Update rules

Все реализованы в одном файле, общий интерфейс — `get_update(x, grad, state)` плюс `reset()` для сброса внутреннего состояния между запусками:

- `SGDRule` — без памяти, просто отдаёт градиент как есть;
- `MomentumRule` — классический momentum со скоростью;
- `NesterovRule` — momentum с поправкой "посмотреть на шаг вперёд";
- `AdaGradRule`, `RMSPropRule`, `AdamRule` — адаптивные методы с покомпонентным масштабированием шага.

## Step strategies

- `FixedStep` — постоянный шаг;
- `DecayingStep` — затухающий шаг вида `initial / (1 + decay * iteration)`;
- `BacktrackingStep` — line search с условием Армихо: уменьшаем шаг до тех пор, пока не выполнится условие достаточного убывания, либо пока не упрёмся в `min_step`.

## Stopping criteria

Оптимизатор принимает список критериев и останавливается по первому сработавшему:

- `MaxIterationsCriterion` — просто лимит на число итераций (и единственный критерий, при котором `success = False`);
- `GradientNormCriterion` — норма градиента упала ниже допуска;
- `ArgumentChangeCriterion` — `x` почти не меняется между итерациями;
- `FunctionChangeCriterion` — значение функции почти не меняется.

## Regularizers

Применяются к градиенту перед update rule, через `apply(x, grad)`:

- `NoRegularization` — заглушка, ничего не делает (используется по умолчанию, если регуляризатор не передан);
- `L2Regularizer`, `L1Regularizer`, `ElasticNetRegularizer` — добавляют к градиенту соответствующий штраф.

## Контейнеры данных

- `OptimizationState` — снимок одной итерации: `x`, `f_val`, `grad`, `grad_norm`, `step_size`;
- `OptimizationHistory` — то же самое, но накопленное по всем итерациям, плюс `param_change_norms` и (опционально) `trajectory`;
- `OptimizationResult` — финальный отчёт о прогоне: лучшая и финальная точка/значение, причина остановки, все счётчики обращений, время работы, история.

## Тестовые функции и лоссы

Тестовые функции (`Ellipsoid`, `Beale`, `Himmelblau`, `Rastrigin`, `Ackley`) — стандартный набор для проверки оптимизаторов, у каждой свои `tags` и известный `global_minimum_x`, чтобы потом сверять найденную точку с эталоном.

`MSE` и `LogLoss` — те же `Objective`, только обёрнутые вокруг реальных данных (`X`, `y`), с добавленным столбцом единиц под свободный член и поддержкой мини-батчей через `indices`.

## Бенчмарки

`RunMetrics` — техническая часть прогона: время, память, счётчики обращений.

`GradientBenchmarkResult` — для проверки самого градиента: насколько он отличается от эталонного (по `L2` и `L∞`), на каком методе и какой размерности это мерили.

`OptimizerBenchmarkResult` — результат конкретной связки (update rule + step strategy + regularizer + гиперпараметры) на конкретной задаче: чем закончилось, сколько итераций, какой `stop_reason`, плюс те же `RunMetrics`.
