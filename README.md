# Optimization Framework

В этом задании необходимо реализовать модульный фреймворк для итеративной градиентной оптимизации, а также набор вспомогательных сущностей: тестовые функции, функции потерь и инструменты для бенчмаркинга получившихся оптимизаторов.

## Основные особенности

- Оптимизатор строится из независимых, взаимозаменяемых компонентов (update rule, step strategy, stopping criteria, regularizer, gradient estimator) — он не должен знать о конкретной реализации каждого из них.
- Поддерживается как full-batch, так и mini-batch режим: целевая функция принимает опциональный набор индексов `indices`, по которым считается значение/градиент.
- Любая целевая функция ведёт собственную статистику обращений: количество вызовов значения, количество вызовов градиента и суммарное количество "просмотренных" объектов (`samples_seen`). Эта статистика должна сбрасываться перед каждым новым запуском оптимизации.
- Процесс оптимизации завершается по дизъюнкции произвольного набора критериев остановки, а не по одному жёстко заданному условию.
- Опционально сохраняется полная история оптимизации (значения функции, нормы градиента, размеры шага, нормы изменения аргумента, траектория) — для последующей визуализации и анализа.

## Вспомогательные классы

### `Objective`

Абстрактный базовый класс для любой оптимизируемой функции (тестовая функция, лосс модели и т.п.). Помимо вычисления значения и градиента, отвечает за подсчёт статистики обращений.

- `__call__(x, indices=None)` — вычислить значение в точке `x`, увеличить счётчик вызовов функции и `samples_seen`;
- `grad(x, indices=None)` — вычислить градиент, увеличить счётчик вызовов градиента и `samples_seen`;
- `_compute_value(x, indices=None)`, `_compute_grad(x, indices=None)` — абстрактные методы, которые должны быть переопределены в наследниках;
- `reset_counters()` — сбросить все счётчики обращений;
- `tags` — множество текстовых меток, описывающих свойства функции (например, `"convex"`, `"multimodal"`, `"ill-conditioned"`).

### Gradient Estimator

Сущность, отвечающая за получение градиента в текущей точке — будь то точный аналитический градиент объекта `Objective` или его оценка (например, по мини-батчу или конечными разностями). Единственный обязательный метод:

- `estimate(objective, x)` — вернуть (возможно, приближённый) градиент `objective` в точке `x`.

### Update Rule

Отвечает за то, как градиент (после регуляризации) превращается в направление обновления параметров. Обязан поддерживать:

- `get_update(x, grad, state)` — получить направление обновления;
- `reset()` — сбросить внутреннее состояние (накопленные моменты, скорость и т.п.) перед новым запуском.

Требуется реализовать как минимум: `SGDRule`, `MomentumRule`, `NesterovRule`, `AdaGradRule`, `RMSPropRule`, `AdamRule`.

### Step Strategy

Отвечает за выбор размера шага на каждой итерации:

- `get_step(function, state)` — вернуть текущий размер шага.

Требуется реализовать как минимум: `FixedStep`, `DecayingStep`, `BacktrackingStep` (линейный поиск с условием Армихо).

### Stopping Criterion

Отвечает за решение об остановке оптимизации. Оптимизатор должен поддерживать произвольный набор критериев, останавливаясь, если выполнился хотя бы один из них:

- `check(state, history)` — правда ли, что критерий остановки выполнен;
- `reason()` — текстовое описание причины остановки.

Требуется реализовать как минимум: `MaxIterationsCriterion`, `GradientNormCriterion`, `ArgumentChangeCriterion`, `FunctionChangeCriterion`.

### Regularizer

Применяется к градиенту перед передачей в update rule:

- `apply(x, grad)` — вернуть регуляризованный градиент.

Требуется реализовать как минимум: `NoRegularization`, `L1Regularizer`, `L2Regularizer`, `ElasticNetRegularizer`.

## Методы `MasterOptimizer`

#### Конструктор

`MasterOptimizer(objective, gradient_estimator, update_rule, step_strategy, stopping_criteria, regularizer=None, store_trajectory_if_dim_leq=2)` — собрать оптимизатор из переданных компонентов. Если `regularizer` не передан, используется `NoRegularization`. Траектория сохраняется только если размерность задачи не превышает `store_trajectory_if_dim_leq`.

#### `optimize(x0)`

Запустить процесс оптимизации из точки `x0` и вернуть `OptimizationResult`. На каждой итерации должно происходить:

1. вычисление текущего значения функции и градиента (через `gradient_estimator` и `regularizer`);
2. фиксация состояния (`OptimizationState`) и проверка всех критериев остановки;
3. при выполнении хотя бы одного критерия — завершение с соответствующим `stop_reason` и итоговой статистикой;
4. иначе — вычисление шага и направления обновления, обновление параметров, запись итерации в историю и обновление наилучшего найденного значения (`best_x`, `best_f`).

## Контейнеры данных

#### `OptimizationState`

Снимок состояния оптимизатора на конкретной итерации: `iteration`, `x`, `f_val`, `grad`, `grad_norm`, `step_size`.

#### `OptimizationHistory`

Накопленная по ходу оптимизации история: `iterations`, `f_values`, `grad_norms`, `step_sizes`, `param_change_norms`, `trajectory` (заполняется только если `store_trajectory = True`).

#### `OptimizationResult`

Итог одного запуска оптимизации: количество итераций, флаг успеха, причина остановки, счётчики обращений к целевой функции (`function_calls`, `gradient_calls`, `samples_seen`), лучшая и финальная точки/значения, финальная норма градиента, история, время работы.

## Бенчмаркинг

### Тестовые функции

Набор классических многомерных тестовых функций, унаследованных от `Objective`, каждая со своим множеством `tags` и известным `global_minimum_x`:

- `Ellipsoid` — выпуклая, при большом коэффициенте растяжения помечается как `"ill-conditioned"`;
- `Beale` — невыпуклая, унимодальная, с "изогнутым" рельефом;
- `Himmelblau` — мультимодальная, с несколькими глобальными минимумами;
- `Rastrigin` — мультимодальная, с большим количеством локальных минимумов, обобщается на произвольную размерность;
- `Ackley` — мультимодальная, с характерным плато вокруг глобального минимума, также обобщается на произвольную размерность.

### Функции потерь

Лоссы для типовых задач машинного обучения, также наследуются от `Objective` и поддерживают мини-батчи через `indices`:

- `MSE` — среднеквадратичная ошибка для линейной регрессии;
- `LogLoss` — логистическая функция потерь для бинарной классификации.

Обе должны добавлять фиктивный столбец единиц к матрице признаков (для свободного члена) и предоставлять `get_n_samples()`.

### Контейнеры для результатов бенчмарков

#### `RunMetrics`

Технические метрики одного запуска: время работы (`wall_time_sec`), пиковое и текущее потребление памяти, количество вызовов функции/градиента, количество просмотренных объектов.

#### `GradientBenchmarkResult`

Результат проверки корректности и производительности вычисления градиента: имя метода, размерность, использованный `epsilon` (для численного дифференцирования, если применимо), ошибки по `L2`- и `L∞`-норме относительно эталонного градиента, `RunMetrics`.

#### `OptimizerBenchmarkResult`

Результат прогона конкретной комбинации (update rule + step strategy + regularizer + гиперпараметры) на конкретной задаче: итоговое и наилучшее значение функции, финальная норма градиента, количество итераций, флаг успеха и причина остановки, `RunMetrics`.
